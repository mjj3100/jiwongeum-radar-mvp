# SETUP.md — 사주 MVP (오행포켓형) 빌드 명세서

> **이 파일을 읽는 대상: Claude Code(AI).**
> 사람(수강생)은 이 파일을 자기 프로젝트 루트에 넣고 "이 SETUP.md대로 만들어줘"라고 지시한다.
> 이 문서는 검증된 빌드 흐름을 그대로 재현해 시행착오를 줄이기 위한 것이다.
> 각 단계에는 **[사람이 직접]** / **[Claude Code가]** 표시가 있다. 표시를 지켜라.

## 0. 이 프로젝트가 만드는 것 (한 줄 요약)

광고 → **리틀리(외부 결제)** → 결제완료 안내 → **웹 회원가입 + 주문번호 입력 → 즉시 자동승인** → AI 분석/멘토 이용.

> 사주(오행) 분석이 기본 예제지만, **분석 도메인만 갈아끼우면** 타로·MBTI·작명·운세 등 어떤 "AI가 개인화 결과를 생성하는 단판 디지털 상품"에도 그대로 적용된다. 큰 틀(결제 동선·인증·승인)은 고정, 도메인 로직만 교체한다.

### 왜 리틀리인가 (설계 전제 — 절대 바꾸지 말 것)
- 일반 PG(토스페이먼츠 등) 직접 연동은 **사업자 심사가 오래 걸리거나 반려**될 수 있다.
- **리틀리는 PG 심사 없이 즉시 결제 링크를 만들 수 있다.** 그래서 MVP 빠른 검증에 적합하다.
- 단, 리틀리는 **서버와 실시간 연동(웹훅)이 없다.** 그래서 "고객이 주문번호를 손으로 옮겨 적는" 구조로 설계한다. 이게 이 명세의 핵심이다.

## 2. 고객 동선 (이 순서를 코드로 강제하라)

```
① 광고  →  ② 리틀리 결제페이지에서 결제
   ↓ (결제 시점엔 아직 웹 회원이 아님 — 가입은 결제 "후")
③ 리틀리 결제완료 화면 + 카카오 알림톡에 "주문번호 16자리" + "주문자 이메일" 표시
   ↓
④ 결제완료 안내(리틀리 '전달 메세지' / PDF)에 있는 가입 링크 클릭 → 웹 /signup
   ↓
⑤ 회원가입(이메일+비번) + 주문번호 입력  →  claim_order RPC 즉시 자동승인
   ↓
⑥ /result 로 직행 → 개인정보 입력 → AI 분석 결과 + (선택)부적 + AI 멘토
```

### 핵심 설계 규칙
- **회원가입은 결제 "후"** 다. 광고가 웹이 아니라 리틀리로 직접 가기 때문. (`가입→결제` 순서로 만들지 마라.)
- **주문번호 = 결제 증거.** 리틀리와 서버가 연동 안 되므로, 고객이 결제완료 화면/알림톡에 뜬 16자리를 가입 시 직접 입력한다.
- **구글폼·Apps Script는 쓰지 않는다.** 웹 `/signup` 화면에서 주문번호를 직접 받는다. (초기 설계에서 구글폼을 검토했으나, 웹에서 직접 받는 게 단계가 적고 이탈이 낮아 제거함.)
- **입력 최소화.** 주문번호 1칸만. 결제이메일 이중검증은 넣지 않는다(이탈 방지). 단, 주문번호 형식검증(숫자 16자리, 공백·하이픈 자동제거, 자릿수 체크)은 넣는다.

## 2-A. 사주 엔진 참고 저장소 (도메인=사주일 때 필수)

사주(만세력) 계산은 **천문학적 음력 변환·절입시각 처리가 필요해 직접 구현하면 정확도 검증이 어렵다.** 아래 검증된 오픈소스를 참고해 핵심 엔진을 설계하라. (Claude Code는 이 저장소들의 계산 로직·데이터 구조를 참고해 만세력→사주팔자→오행 분포 계산 모듈을 만든다. 라이선스를 확인하고 준수할 것.)

- **사주/만세력 라이브러리 (1순위 권장)**: https://github.com/golbin/ssaju
  - TypeScript, 의존성 0개, gzip ~15KB. 한국천문연구원 음력 데이터 + 절입시각 기반 정확 계산.
  - 원국·십성·12운성·합충형파해·신살·격국·용신·대운·세운·월운·공망 등을 **LLM friendly하게 압축 제공** → Claude 프롬프트에 그대로 넣기 좋다.
  - 본 스택(Next.js/TS)과 가장 잘 맞으므로 **사주 계산 엔진의 기본 후보**로 삼아라.
- **사주 계산 참고**: https://github.com/rath/orrery
- **만세력 데이터/계산 참고**: https://github.com/yhj1024/manseryeok

> 활용법: 위 저장소를 클론하거나 README/소스를 읽어 **(생년월일시 → 사주팔자 8자 → 오행(목화토수금) 분포)** 까지 계산하는 모듈을 만든다. 그 오행 분포(부족/과다 오행)를 `/api/analyze`의 Claude 프롬프트에 넣어 개인화 리포트를 생성한다. 반려동물 페어 분석을 한다면 보호자·반려동물 각각 계산 후 두 오행을 비교한다.
> ⚠️ 사주 계산 정확도는 **게이트(Gate)** 로 검증하라: 알려진 생년월일 몇 건의 사주팔자가 기존 만세력 서비스와 100% 일치하는지 확인 후 다음 단계로.

## 3. 데이터베이스 스키마

### 3-1. 핵심 테이블 (Claude Code가 마이그레이션 0001로 작성)

최소 구성:
- `profiles` — 사용자. `id(uuid)`, `email`, `status(user_status enum: pending|approved)`, 생성 트리거로 가입 시 `pending` 생성.
- `orders` — 주문번호 원장. `order_no(text)`, `payment_email`, `product`, `amount`, `claimed_by`, `claimed_at`.
- `entitlements` — 이용권. `user_id`, `product`, `mentor_quota(int)`.
- (도메인) `results` / `mentor_messages` 등 — 분석 결과 저장, 멘토 대화 기억(교차세션). RLS로 **사용자별 격리**.

**RLS 필수**: 모든 사용자 데이터 테이블에 Row Level Security를 켜고, "본인 것만" 정책을 건다. (동명이인/타인 데이터 접근 사고 구조적 방지.)

### 3-2. 주문번호 1회용 자동승인 (마이그레이션 0002 — claim_order)

"결제 1건 = 승인 1회"를 **DB 레벨에서 원자적으로** 보장한다. 핵심:
- `orders.order_no` 에 **UNIQUE 제약** → 같은 주문번호 두 번째 사용 시 `unique_violation` → `order_used` 반환.
- `claim_order(p_order_no, p_signup_email, p_payment_email)` RPC, `security definer`.
  - 반환값: `approved | order_used | no_account | already_approved | no_order`
  - 흐름: 주문번호 공백/빈값 체크 → 가입 이메일로 사용자 조회 → 이미 승인이면 종료 → `orders`에 insert 시도(중복이면 `order_used`) → `profiles.status='approved'` 업데이트 + `entitlements` 부여 → `approved`.
- 권한: `revoke all ... from public, anon, authenticated` 후 `grant execute ... to service_role`. **일반 사용자가 직접 호출해 자가 승인하는 것 차단.**

> 분석 결과를 위한 `results`, 멘토 대화 `mentor_messages`도 RLS로 승인 사용자만 쓰기 가능하게.

### 3-3. ⚠️ 스키마 적용 시 사람이 자주 막히는 지점 (Claude Code는 사람에게 미리 안내하라)

- 스키마 SQL을 **채팅에 붙여 복사하면 중간이 잘린다.** → "코드블록 우측 상단 **복사 버튼**으로 통째 복사"하라고 안내. 길면 (A)테이블 변경 / (B)함수 두 덩어리로 나눠서 각각 별도 코드블록으로 제공.
- Supabase SQL Editor 실행 시 **"Potential issues detected"(destructive/RLS) 경고**가 뜬다 → 새 테이블엔 **"Run and enable RLS"**, 기존 테이블 수정엔 그대로 Run. 정상이다.
- 두 번째 실행 시 `relation "..." already exists (42P07)` → **에러가 아니라 이미 적용됐다는 뜻.** 무해. 함수 존재는 `select proname from pg_proc where proname='claim_order';` 로 확인.
- 성공 메시지는 `Success. No rows returned` (테이블/함수 생성은 반환 행이 없는 게 정상).

### 3-4. Supabase 인증 설정 (사람이 직접)
- Authentication → Providers → Email → **"Confirm email" OFF** (가입 즉시 로그인, MVP 테스트 편의).
- 배포 후: Authentication → URL Configuration → **Site URL = 배포 주소**.
- Redirect URLs: 이메일/비번 로그인만 쓰면 **불필요**. 소셜 로그인/비번 재설정 붙일 때만 `배포주소/**` 추가.

## 4. 페이지/엔드포인트 (최소 범위 — 늘리지 말 것)

| 경로 | 역할 | 접근 |
|------|------|------|
| `/` | 랜딩(핵심 카피 + 리틀리 결제 CTA) | 공개 |
| `/signup` | 결제 후 가입. **주문번호 입력칸 + 이메일 + 비번** → claim_order 즉시 승인 → /result | 공개 |
| `/login` | 로그인 | 공개 |
| `/result` | 개인정보 입력 → AI 분석 결과 + 부적 + 멘토 | **승인(approved)만** |
| `/try` (선택) | 결제 없이 분석 체험(데모/홍보용) | 공개 — **출시 전 제거/제한 필수** |
| `/api/analyze` | 분석 생성 (Claude 호출) | 서버 |
| `/api/mentor` | 멘토 채팅 스트리밍 | **인증 필수(무인증 401)** |

### 보안 게이트 (반드시)
- `/api/mentor` 등 **비용이 드는 LLM 엔드포인트는 무인증 시 401.** (비용 폭탄/무료 프록시 악용 차단.)
- `/result` 는 승인 사용자만. 미인증은 로그인으로 리다이렉트.
- `/try`(무료 풀분석)는 **출시 전 반드시 제거하거나 결과 일부만** 노출. 안 그러면 결제 안 한 사람이 공짜로 다 본다.
- 멘토 입력: **전송 중 입력 잠금(동기 가드)** + **한글 IME 조합 중 Enter 무시**(이중 전송 방지).

## 5. 환경변수 (.env.local)

```
NEXT_PUBLIC_SUPABASE_URL=https://<project>.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=sb_publishable_...   # 공개 가능(브라우저 노출 OK)
SUPABASE_SERVICE_ROLE_KEY=sb_secret_...            # 절대 비밀. 서버 전용. RLS 우회
ANTHROPIC_API_KEY=sk-ant-api03-...                 # 비밀
NEXT_PUBLIC_LITTLY_URL_PAIR=https://litt.ly/<slug> # 결제 링크
```

### ⚠️ 키 관련 사람이 막히는 지점 (Claude Code는 사람에게 안내)
- Supabase 신버전 UI는 **`sb_publishable_` / `sb_secret_`** 형태다(구버전 JWT `eyJ...`가 아님). 새 프로젝트엔 publishable/secret만 보일 수 있다. `@supabase/ssr`과 호환되니 그대로 쓴다.
- `SUPABASE_SERVICE_ROLE_KEY`(secret)는 **Settings → API Keys → Secret keys 섹션에서 Reveal/복사**. 안 보이면 스크롤/“Create new secret key”.
- Anthropic 키는 **생성 직후 1회만** 전체 표시 → 즉시 복사. **Billing 크레딧 충전(최소 약 $5)** 안 하면 호출 시 잔액 오류.
- **`.env.local` 은 절대 git에 올리지 않는다.** `.gitignore`에 포함 확인. secret 키가 GitHub에 올라가면 Supabase가 자동 무효화한다.

## 6. 빌드 순서 (Claude Code가 따를 권장 순서)

> 원칙: **결제와 무관한 영역(인증·분석·멘토)부터** 완성하고, 결제 연동은 마지막에. (사람이 키/계정을 준비하는 동안 병렬 진행 가능.)

1. **[사람이 직접]** Supabase 프로젝트 생성 + 키 3개, Anthropic 키 1개 발급, 크레딧 충전.
2. **[사람이 직접]** `.env.local` 작성. (Claude Code는 값을 채팅으로 받지 말 것 — 사람이 파일에 직접 입력.)
3. **[Claude Code]** Next.js + Tailwind 골격, 랜딩, /signup, /login.
4. **[Claude Code]** 도메인 분석 모듈(예: 사주 — §2-A 참고 저장소로 만세력→사주팔자→오행 분포 계산) + `/api/analyze`(Claude 프롬프트) + /result 렌더. **사주 계산 정확도 Gate 통과 확인.**
5. **[Claude Code]** AI 멘토 `/api/mentor`(스트리밍, 무인증 401, 교차세션 기억).
6. **[Claude Code]** 스키마 0001(profiles/orders/entitlements/results/mentor_messages + RLS) 작성 → **[사람이 직접]** SQL Editor 실행.
7. **[Claude Code]** 스키마 0002(claim_order) 작성 → **[사람이 직접]** 실행.
8. **[Claude Code]** /signup 에 주문번호 입력 + claim_order 연결 + 형식검증. **결제 후 가입 동선**으로 구성.
9. **[사람이 직접]** 리틀리 상품 등록 → 결제 링크 확보 → `.env.local`/CTA에 연결.
10. **[Claude Code]** 보안 리뷰(인증·결제·승인·접근제어) → 빌드 통과 확인.
11. **[사람+Claude Code]** 배포(§7).
12. **[사람이 직접]** 리틀리 전달 메시지/PDF에 가입 링크(§8) → **실결제 1건 end-to-end 테스트(§9).**

## 7. 배포 (Vercel, GitHub 연동)

1. **[Claude Code]** git 저장소 상태 확인 → **`.env.local`이 추적되지 않는지 정밀 확인(가장 중요)** → GitHub **비공개** 저장소 생성 + 푸시.
2. **[사람이 직접 — 강력 권장] 키 회전:** 빌드/검수 과정에서 노출됐을 수 있는 **service_role 키와 Anthropic 키를 재발급(Rotate)** 하고 `.env.local` 갱신. (URL/anon은 공개값이라 회전 불필요.)
3. **[사람이 직접]** vercel.com → Add New → Project → 그 저장소 Import.
4. **[사람이 직접] 환경변수 5개 등록** (회전한 새 키 사용). `.env.local`은 Vercel에 자동 반영되지 않는다.
5. Deploy → 1~2분 후 발급.
   - ⚠️ `xxxx-랜덤-projects.vercel.app` = **임시 배포 주소**(배포마다 바뀜, 고객에게 주지 말 것).
   - ✅ 고객용은 **고정 대표 주소**: 대시보드 **Domains**의 `<project>.vercel.app`.
6. **[사람이 직접]** Supabase Site URL = 대표 주소.
7. 이후 `git push` 시 Vercel **자동 재배포**(CI/CD).

> Vercel 로그인 중 **2FA 설정 화면**이 떠도 배포 필수 아님 → "Skip" 가능(출시 후 보안 강화로 등록 권장).

## 8. 리틀리 설정 (사람이 직접 — Claude Code가 못 보는 영역)

리틀리 상품 편집에서 **"전달 메세지"**(결제완료 안내, 자유 편집 가능)를 가입 안내로 교체한다. 기존 "오픈채팅/생년월일 수동전송" 안내가 있으면 **제거**한다(웹 입력폼이 대체).

전달 메세지 예시:
```
구매해주셔서 감사합니다 🐾

아래 링크에서 회원가입 후, 결제완료 화면(또는 카카오 알림톡)의
'주문번호 16자리'를 입력하면 즉시 승인되어 바로 이용하실 수 있어요.

👉 https://<배포 대표주소>/signup

개인정보(생년월일 등)는 가입 후 웹에서 직접 입력합니다.
※ 주문번호는 1회만 사용 가능 · 본 서비스는 엔터테인먼트 콘텐츠입니다.
```

- 첨부 PDF(선택): "시작 안내서" 용도로 위 3단계(가입 링크·주문번호 입력·분석 시작)를 담아 결제완료 화면에 첨부.
- 주문번호는 **결제완료 화면 + 카카오 알림톡** 두 곳에 표시된다(고객이 못 찾을 위험 ↓). 형식은 **숫자 16자리** 고정 → §4 형식검증과 일치.

## 9. 실결제 검수 (사람이 직접 — 진짜 종착점)

테스트 주문번호가 아니라 **실제 결제로 발급된 진짜 주문번호**로 한 바퀴 돌린다.

1. 리틀리에서 본인 카드로 결제 → 진짜 주문번호 16자리 + (선택)PDF 수신.
2. 전달 메세지의 `/signup` → 가입 + 주문번호 입력 → **즉시 /result 이동 = 자동승인 성공**.
3. 개인정보 입력 → 분석 결과 + (선택)부적 + 멘토 응답 정상.
4. 같은 주문번호로 또 가입 시도 → **"이미 사용된 주문번호" 차단 = 1회용 정상**.
5. 리틀리 [수익관리]→[판매]에서 테스트 결제 **환불**.

> 가입 후 /result가 아니라 "주문번호 확인 실패"가 뜨면 → 실제 주문번호 형식(자릿수/문자 포함 여부)을 확인하고 §4 형식검증을 실제 형식에 맞춘다.

## 11. 다른 도메인으로 응용하기

큰 틀(리틀리 결제 → 결제 후 가입 → 주문번호 1회용 승인 → AI 결과/멘토)은 **그대로 유지**하고 아래만 교체:
- **분석 모듈**: 사주 만세력/오행(§2-A 참고 저장소 활용) → (타로 카드 로직 / MBTI 채점 / 작명 알고리즘 등)
- **`/api/analyze` 프롬프트**: 도메인 톤·출력 형식
- **/result 렌더**: 결과 카드 구성(부적 등 부가물은 선택)
- **상품 가격/문구/리틀리 상품명**
- 결제·인증·승인 코드(`claim_order`, RLS, 게이트)는 **재사용**한다.

## 부록. 이 흐름에서 사람이 가장 많이 막히는 5가지 (Claude Code는 선제 안내)

1. **스키마 SQL 잘림** → 복사 버튼으로 통째 복사, 길면 A/B 분할.
2. **`already exists` 에러** → 실패 아님, 이미 적용됨. `pg_proc`로 함수 확인.
3. **주문번호 출처** → 리틀리 결제완료 화면 + 카카오 알림톡(16자리). 서버 연동 없이 고객이 옮겨 적는 구조.
4. **회원가입 순서** → 광고가 리틀리로 직행하므로 "결제 → 가입". 일반 SaaS와 반대.
5. **임시 vs 대표 배포 주소** → 리틀리엔 반드시 고정 대표 주소(`<project>.vercel.app`).
