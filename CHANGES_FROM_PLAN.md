> 이 문서는 "기획안·전자책에 있던 원안"과 "실제로 대화하며 개발해 온 것"의 간극을 추적하는 용도다.
> Fable(Claude 웹)에서 향후 방향을 논의할 때 이 문서를 통째로 붙여넣고 시작하면 된다.

## 1. 스코프 축소 — 5단계 구독 모델 → 사실상 1개 상품

기획안 최종 비전: **Free → Starter → Builder → Diagnostic/Deep → Season Pass** 구독+단건 이중 매출 SaaS.

MVP 실제 구현: 처음엔 **bundle(전자책+진단리포트 결합 단건) + starter(월 9,900원 구독)** 2개 상품으로 축소했으나, 아래 §3의 판단에 따라 **Starter를 랜딩에서 잠정 비노출** 처리해 현재는 사실상 **bundle 단일 상품**으로 운영 중이다(코드는 유지, 노출만 플래그로 차단).

**백로그로 이동(미구현):**
- Builder(신청준비방 7탭)
- Diagnostic/Deep 단건 결제
- Season Pass
- D-day 플래너, 사업계획서 작성 보조 전체 (위험문장 검사만 1차로 포함)
- "AI 멘토"(교차세션 대화형) — 사주 도메인 개념이라 이 도메인에 대응 기능 없음. Starter 대시보드의 "맞춤 공고 재조회"로 대체, `/api/mentor` 자체를 제외

**근거:** `spec.md` §1, `todo.md` 백로그 섹션

## 2. 결제·주문 처리 방식 — SETUP.md 원안에서 변경

**원안(SETUP.md):** 고객이 입력한 주문번호를 그 자리에서 `insert`하고 UNIQUE 제약으로 중복을 방지.

**실제 구현:** 상품이 2종(bundle/starter)이고 Starter는 재구매(구독 연장)가 있어, **운영자가 결제 알림을 보고 `orders` 테이블에 주문번호를 사전 등록**(product 태그 포함)하는 방식으로 변경. `claim_order` RPC는 "존재하는 미클레임 주문번호에 `claimed_by`를 채워 넣는" 방식으로 재설계(v2). Starter는 `expires_at`을 `greatest(now(), 기존만료일) + 30일`로 연장.

**부작용(1차):** 주문번호 사전 등록을 운영자가 수동으로 해야 함(당시는 Claude Code가 스크립트로 대행) — 결제량이 늘면 리틀리 알림 이메일 파싱 자동화가 필요(백로그).

**추가 진화(커밋 `c5a820e`, 7/7):** 결제 직후 아직 운영자가 주문번호를 등록하기 전에 고객이 가입을 시도하면 실패하던 문제를 해결하기 위해, **가입과 주문번호 클레임을 분리**했다.
- 가입 시 주문번호가 `orders`에 아직 없어도 가입 자체는 통과시키고 `/pending`으로 보낸다.
- `/pending`은 방문할 때마다 자동으로 재클레임을 시도한다 — 운영자가 그 사이 등록해두면 사용자는 아무 조작 없이 새로고침만으로 풀린다.
- **`/admin/orders` 관리자 콘솔을 신규 구축**: 관리자 이메일로 로그인하면 주문번호+상품종류를 웹 폼으로 직접 등록/삭제(환불)할 수 있다. 이는 원 기획안에는 전혀 없던 "운영자 셀프서비스 도구"로, Claude Code가 매번 스크립트를 대행하던 방식에서 사람이 직접 운영하는 방식으로 전환한 것이다.
- 환불(주문번호 삭제) 시 해당 계정의 이용권도 함께 회수하고, 남은 이용권이 없으면 `profiles.status`를 다시 `pending`으로 되돌려 재구매 시 `/result`↔`/pending` 무한 리다이렉트가 나던 버그도 수정(커밋 `9d0c6d3`).

**근거:** `spec.md` §3-2, §3-7 "SETUP.md와의 차이" 단락, 커밋 `c5a820e`, `9d0c6d3`, `src/app/pending/`, `src/app/admin/orders/`

## 3. Starter 구독 — 구조적 한계 확인 후 랜딩 노출 잠정 중단 (방향 전환)

리틀리가 정기결제를 지원하지 않아 "매달 재구매 + 주문번호 재입력" 방식으로 구현했고, 일반 SaaS 구독 UX 대비 이탈률이 높을 수 있는 구조적 리스크를 안고 출시했다(여기까지는 기존 기록과 동일).

**새로운 판단(커밋 `c5a820e`, `src/lib/constants.ts`):** 실제 운영을 해보며 "무제한 맞춤공고 재조회"라는 Starter의 핵심 가치 제안이 약하다고 판단해, **랜딩 페이지에서 Starter 노출을 `SHOW_STARTER_ON_LANDING = false` 플래그로 잠정 중단**했다. 이용권 로직(엔터틀먼트, `claim_order` 구독 연장 등) 자체는 코드에 그대로 남아 있어 플래그만 켜면 즉시 재노출 가능한 상태다.

→ **이것이 이번 기획안 대비 가장 큰 "방향" 변화**다. 기획안은 구독을 5단계 중 하나의 축으로 설계했지만, 실제로는 "구독형 재조회"가 이 도메인(지원사업 매칭)에서 반복 결제를 정당화할 만큼 매력적이지 않다는 가설이 생긴 상태 — Fable 세션에서 다룰 만한 핵심 논점.

**근거:** `spec.md` §9, 커밋 `c5a820e`, `src/lib/constants.ts`

## 15. Fable(웹) 세션에서 다룰 만한 방향 논점

기획안·전자책에는 없었지만 실제로 만들고 운영하며 새로 생긴 질문들 — 다음 방향을 잡을 때 출발점으로 쓸 것.

1. **Starter 구독을 살릴지, 접을지, 다른 형태로 바꿀지.** §3에서 "무제한 재조회"의 가치가 약하다고 판단해 랜딩 노출을 껐다. 완전 폐기 / 가치 제안 재설계(예: Diagnostic 단건으로 전환) / 리틀리 정기결제 지원 시 재검토, 세 갈래 중 방향을 정해야 한다.
2. **가격 미확정 상태가 장기화되고 있다.** bundle 판매가가 기획안에도, 지금 코드에도 없다(`null`). 정확도 검증(§12)까지 끝났으므로 가격만 남은 상태 — 얼마를 받을지 결정이 필요.
3. **운영 방식이 "Claude Code가 스크립트로 대행"에서 "사람이 웹 콘솔로 직접 운영"으로 굳어졌다(§2, §13).** 이게 결제량이 늘어도 버틸 방식인지, 아니면 리틀리 알림 이메일 파싱 자동화(백로그)로 넘어가야 할 타이밍인지 판단이 필요.
4. **기획안의 5단계 중 Free/Builder/Diagnostic-Deep/Season Pass는 전혀 손대지 않은 백로그 상태다(§1).** MVP가 bundle 단일 상품으로 수렴한 지금, 다음 상품을 어떤 순서로 붙일지(혹은 이대로 단일 상품에 집중할지) 결정이 필요.
5. **"AI 멘토"(사주 도메인 개념)는 이 도메인에 대응 기능이 없다고 스코프에서 제외했다(§1).** 지원사업 도메인에서 이 역할을 대체할 게 정말 필요 없는지, 아니면 다른 형태(예: 서류 작성 보조, D-day 알림)로 필요성이 다시 떠오르는지 재검토할 만하다.
