# 사업화 기획안 대비 MVP 개발 변경사항

> 근거: `지원금 레이더 ppt.pptx`(사업화 기획안), `지원금_레이더_전자책.pdf`(도메인 원천자료), `SETUP.md`(기술 빌드 명세 원안) vs `spec.md`/실제 코드/`todo.md`(실개발 결과)
> 최종 갱신: 2026-07-07 (커밋 `a3b6068` 기준)

## 1. 스코프 축소 — 5단계 구독 모델 → 2개 상품

기획안 최종 비전: **Free → Starter → Builder → Diagnostic/Deep → Season Pass** 구독+단건 이중 매출 SaaS.

MVP 실제 구현: **bundle(전자책+진단리포트 결합 단건) + starter(월 9,900원 구독)** 2개 상품만 구현.

**백로그로 이동(미구현):**
- Builder(신청준비방 7탭)
- Diagnostic/Deep 단건 결제
- Season Pass
- D-day 플래너, 사업계획서 작성 보조 전체 (위험문장 검사만 1차로 포함)
- "AI 멘토"(교차세션 대화형) — 사주 도메인 개념이라 이 도메인에 대응 기능 없음. Starter 대시보드의 "맞춤 공고 재조회"로 대체, `/api/mentor` 자체를 제외

**근거:** `spec.md` §1, `todo.md` 백로그 섹션

## 2. 결제·주문 처리 방식 — SETUP.md 원안에서 변경

**원안(SETUP.md):** 고객이 입력한 주문번호를 그 자리에서 `insert`하고 UNIQUE 제약으로 중복을 방지.

**실제 구현:** 상품이 2종(bundle/starter)이고 Starter는 재구매(구독 연장)가 있어, **운영자가 결제 알림을 보고 `orders` 테이블에 주문번호를 사전 등록**(product 태그 포함)하는 방식으로 변경. `claim_order` RPC는 "존재하는 미클레임 주문번호에 `claimed_by`를 채워 넣는" 방식으로 재설계(v2). Starter는 `expires_at`을 `greatest(now(), 기존만료일) + 30일`로 연장.

**부작용:** 주문번호 사전 등록을 운영자가 수동으로 해야 함(현재는 Claude Code가 스크립트로 대행 중) — 결제량이 늘면 리틀리 알림 이메일 파싱 자동화가 필요(백로그).

**근거:** `spec.md` §3-2, §3-7 "SETUP.md와의 차이" 단락

## 3. Starter 구독 방식의 구조적 한계

리틀리가 정기결제를 지원하지 않아 "매달 재구매 + 주문번호 재입력" 방식으로 구현. 일반 SaaS 구독 UX 대비 이탈률이 높을 수 있는 구조적 리스크를 안고 출시. 출시 후 이탈률을 보고 리틀리 정기결제 기능 재검토 예정.

**근거:** `spec.md` §9

## 4. 사업 정보 입력 항목 — 9개 → 11개로 확대

기획안 초안(`spec.md` 최초 버전)은 9개 항목(창업상태/지역/업종/설립일/매출구간/직원수/아이템설명/필요지원/준비도)으로 설계했으나, **원래 오픈카톡 상담 프로세스에 있던 나이대·성별 항목을 복원**하여 11개로 확대. 청년/여성/중장년 특화 공고 판별에 사용하도록 Claude 프롬프트에도 반영.

**근거:** `todo.md` Phase 2, 커밋 `afb3150`, `src/app/result/BusinessProfileForm.tsx`

## 5. 규칙 기반 1차 필터 로직 — "소상공인" 정의 구체화

기획안/스펙 초안에는 단순히 "업력 요건 초과 시 제외" 정도로만 기술되어 있었으나, 실제 구현 과정에서 **"소상공인"은 사업자 등록 형태(개인/법인)가 아니라 매출·업종 기준 규모 분류**라 개인사업자·법인사업자와 상호배타적이지 않다는 점을 도메인 자료(전자책 이민재 사례)로 확인. 공고가 "소상공인"만 요구해도 개인/법인사업자를 규칙 단계에서 걸러내지 않고 2차 Claude 판정에 위임하도록 로직을 구체화. 예비창업자는 정의상 소상공인일 수 없으므로 계속 필터링.

**근거:** `src/lib/grants/match.ts` 주석, `todo.md` Phase 3

## 6. 매칭 결과 0건을 정상 케이스로 허용 (버그 수정)

응답 스키마가 `matches.min(1)`을 강제해서, Claude가 "정말 맞는 공고가 없다"고 판단한 정상적인 경우에도 스키마 검증 실패로 "분석 중 문제가 발생했습니다" 에러가 잘못 표시되던 버그 발견. `min(1)` 제약을 제거하고 빈 결과 전용 안내 UI(원인 설명 + 재시도 유도)를 추가.

**근거:** 커밋 `a3b6068`, `todo.md` Phase 3

## 7. 공고 원문 길이로 인한 Claude 응답 JSON 잘림 (버그 수정)

실제 공고 원문이 예상보다 길어 Claude 응답이 `max_tokens`에 걸려 JSON이 잘리는 문제 발생. 입력 텍스트 절삭 로직 추가 + `max_tokens`를 8192로 상향하여 해결.

**근거:** 커밋 `7142de8`, `todo.md` Phase 3

## 8. DB 마이그레이션 — 부분 인덱스 → 일반 유니크 제약

`grant_listings`의 유니크 제약을 부분 인덱스(partial index)로 구현했으나, PostgREST의 `onConflict` upsert가 부분 인덱스를 인식하지 못해 일반 유니크 제약으로 교체(마이그레이션 0004).

**근거:** 커밋 `3c441d1`, `todo.md` Phase 4

## 9. 공고 데이터 소스 — 시드 데이터 → 실시간 API 연동 완료

기획안/스펙은 "API 키 발급 대기 중 전자책 수록 공고 4건을 수동 시드로 먼저 검증하고, 발급되면 API로 전환"하는 단계적 계획이었음. 실제로는 K-Startup·기업마당 두 API 키 모두 발급받아 `/api/grants/sync`(Vercel Cron, 매일 03:00 UTC)로 완전 자동화 완료 — 프로덕션에 실제 지원사업 318건(K-Startup 218 + 기업마당 100) 저장 확인. 수동 시드 4건은 폴백용으로만 유지.

**근거:** `todo.md` Phase 6, `src/lib/grants/kstartup.ts`, `src/lib/grants/bizinfo.ts`

## 10. 가격 미확정 (기획안에 없던 갭)

기획안(PPT)에 bundle 상품의 확정 판매가가 없어 `PRICING.bundle.price`를 `null` 플레이스홀더로 남겨둠. Starter는 9,900원으로 기획안대로 확정. 사람이 실제 판매가를 정하면 코드와 리틀리 상품 가격을 함께 갱신해야 함.

**근거:** `src/lib/constants.ts`

## 11. 배포 인프라 이슈 (기획안에 없던 실무 대응)

Vercel이 해당 계정에서 개인(personal) 스코프 배포를 지원하지 않아(`personal_scope_not_allowed`) 팀 멤버십 문제로 배포가 막힘. Pro 유료 업그레이드 대신 **GitHub 저장소를 Public으로 전환**하여 무료로 해결(비밀키는 `.env.local`에 있어 git 미추적 — 노출 없음 확인).

**근거:** `todo.md` Phase 8

## 12. 정확도 검증 결과

전자책 원본 이민재 케이스(관련성22·구체성16·차별성17·실현가능성18=73점)와 실제 프로덕션 재현 결과(22·15·16·17=70점)를 대조 — 거의 일치, 위험문장 탐지도 전자책 패턴과 동일하게 작동함을 확인. 기획안에는 없던 정량적 검증 게이트를 자체적으로 도입해 통과.

**근거:** `todo.md` Phase 3, 커밋 `8060fe0`
